# Решение 0002: runtime Markdown-правила

## Статус

Принято.

## Контекст

Боту нужны предметные правила, которые помогают интерпретировать свободные
запросы менеджеров: когда использовать КОРП-лицензии, какие апгрейды допустимы,
когда предложить готовый бандл и как вести себя при неоднозначностях.

Эти правила должны быть достаточно простыми, чтобы их можно было быстро
добавлять и править без изменения Python-кода. При этом они участвуют в
runtime-поведении бота, поэтому не должны храниться как обычная документация в
`docs/`.

## Решения

- Хранить предметные правила в корневой директории `rules/`.
- Писать правила простым русским Markdown без обязательного DSL, YAML-блоков или
  строгой схемы.
- Использовать файлы правил как runtime-контекст для LLM при обработке запроса.
- Разделить правила по темам:
  - `rules/licensing.md`
  - `rules/upgrades.md`
  - `rules/bundles.md`
  - `rules/quote-behavior.md`
- Оставить SQLite источником состояния диалогов и черновиков.
- Оставить MCP-сервер `mcp-1c-price` источником истины по продуктам и ценам.
- Не вводить полноценную автономную агентную систему в v1.

## Последствия

- Правила можно редактировать как обычный текст и хранить вместе с проектом.
- Бот получает гибкую предметную память без отдельной БД или админки для правил.
- Качество применения правил зависит от LLM и качества промпта, поэтому спорные
  случаи должны приводить к уточняющим вопросам.
- Правила в `rules/` нужно считать частью поведения приложения и покрывать
  ручными или автоматическими сценариями проверки.
- В будущем правила можно заменить на строгий rule engine, БД или админку, если
  свободного Markdown станет недостаточно.
